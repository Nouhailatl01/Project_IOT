## вқ— The Problem

In medical analysis laboratories, biological and chemical samples must be stored at a **strictly controlled temperature between 2 В°C and 8 В°C**. Before this system, the reality looked like this:

| Issue | Impact |
|-------|--------|
| рҹ§‘вҖҚрҹ”¬ Manual checks every 2 hours | Temperature excursions go undetected for up to **120 minutes** |
| рҹ“Ӣ Paper-based logs | No digital traceability, impossible to audit |
| рҹ”• No automated alerts | Staff must physically be present to notice failures |
| рҹ’ё No incident tracking | Costs, root causes, and response times are invisible |

A single undetected temperature excursion can compromise an entire batch of samples вҖ” leading to **invalid results, patient harm, and regulatory non-compliance**.

---

## вң… The Solution

A fully automated IoT monitoring system that:

- **Monitors continuously** вҖ” ESP8266 + DHT11 sensors report every 20 minutes via MQTT
- **Detects instantly** вҖ” Django backend checks thresholds on every incoming reading
- **Alerts automatically** вҖ” Multi-level email escalation triggers when temperature drifts out of range
- **Traces everything** вҖ” Every action, alert, and intervention is logged with timestamp, actor, and result
- **Displays in real-time** вҖ” React dashboard gives operators live visibility at a glance

## вңЁ Key Features

### рҹ”ҙ Real-Time Temperature Monitoring
- Continuous 24/7 surveillance of refrigerator temperature and humidity
- Configurable min/max thresholds per sensor (default: 2 В°C вҖ“ 8 В°C)
- Automatic sensor identification via unique MQTT topic routing

### вҡЎ Intelligent Alert & Escalation System

```
ALERT DETECTED  (T < 2В°C  or  T > 8В°C)
        в”Ӯ
```

Every system event is automatically recorded with:

- **WHEN** it occurred (timestamp + IP address)
- **WHAT** the exact action was
- **RESULT** вҖ” success or error

### рҹҺ« Automated Incident Ticketing (CMMS)
- Tickets auto-created on alert detection
- Assignable to operators with full status tracking
- Comments and intervention history per ticket
- Full incident lifecycle: `OPEN вҶ’ IN PROGRESS вҶ’ CLOSED`

### рҹ“Ҡ React Dashboard
- Live measurement cards (temperature & humidity)
- Historical charts per sensor
- Incident archive with drill-down detail view
- Built-in API tester for simulation and demos

## рҹҡҖ Getting Started

### Prerequisites

- Python 3.8+
- Node.js 18+
- Mosquitto MQTT Broker installed locally
- PostgreSQL (production) or SQLite (development)

### 1. Clone the repository

```bash
git clone https://github.com/your-username/chainfroid-monitoring.git
cd chainfroid-monitoring
```

### 2. Backend setup (Django)

```bash
cd backend
python -m venv venv
source venv/bin/activate          # Windows: venv\Scripts\activate
pip install -r requirements.txt

# Configure environment variables
cp .env.example .env
# Edit .env вҶ’ DATABASE_URL, EMAIL_HOST_USER, EMAIL_HOST_PASSWORD, SECRET_KEY

python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

### 3. Start MQTT Listener

```bash
# In a separate terminal (venv activated)
python manage.py mqtt_listener
```

### 4. Frontend setup (React)

```bash
cd frontend
npm install
npm start
```

### 5. Flash the ESP8266

Open `firmware/esp8266_dht11.ino` in Arduino IDE, then update:

```cpp
const char* ssid        = "YOUR_WIFI_SSID";
const char* password    = "YOUR_WIFI_PASSWORD";
const char* mqtt_server = "YOUR_LOCAL_IP";   // e.g. 192.168.1.10
const int   mqtt_port   = 1883;
```

Upload to your ESP8266 NodeMCU board and monitor via Serial (115200 baud).

```python
# Simplified pseudocode вҖ” see backend/alerts/services.py for full implementation

def handle_new_reading(temperature, humidity, capteur):
    if temperature < capteur.seuil_min or temperature > capteur.seuil_max:
        alerte = create_alert(temperature, type="CRITICAL")
        ticket  = create_ticket(alerte)
        log_audit_event("ALERT_CREATED", alerte)

        # Escalation schedule (Celery tasks)
        send_email(operators[0])                            # T+0 min
        schedule(send_email(operators[:2]), delay=4*60)    # T+4 min
        schedule(send_email(operators[:3]), delay=7*60)    # T+7 min
    else:
        save_reading(temperature, humidity)                # Normal вҖ” store only
```

## вҡ пёҸ Limitations & Future Work

### Current Limitations

| Limitation | Detail |
|---|---|
| DHT11 precision | Вұ2 В°C accuracy вҖ” may be insufficient for ISO-certified medical environments |
| Wi-Fi dependency | Connectivity loss = gap in monitoring coverage |
| Single-site only | Currently deployed for one laboratory |
| No offline buffer | Readings missed during connectivity outages are not recovered |
